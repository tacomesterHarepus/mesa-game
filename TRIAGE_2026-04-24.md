## Item 1 — Cards still not drawn (Category A)

**User report:** Gave AI +2 RAM via allocation, expected 6 cards, saw 4. Claimed fix from earlier today.

**Why the earlier fix didn't work — two separate gaps:**

### Gap A: resolve-next-virus copy of advanceTurnOrPhase was never updated

`resolve-next-virus` and `end-play-phase` each maintain a **separate copy** of `advanceTurnOrPhase` (commented "duplicated for independent deployment"). When the Bug 2 fix added `drawCardsForPlayer` calls to `end-play-phase`, the identical function in `resolve-next-virus` was never touched.

Turn advancement flows:
- End turn with **0 viruses** → `end-play-phase` calls its `advanceTurnOrPhase` (has draw) ✓
- End turn with **≥1 viruses** → `end-play-phase` sets `phase=virus_resolution`, returns. Later `resolve-next-virus` is called, exhausts the queue, then calls **its own** `advanceTurnOrPhase` (no draw) ✗

The user had CPU=2, played 2 cards → 1 virus generated → went through `resolve-next-virus` path → no draw.

### Gap B: allocate-resources never draws for the first player of round 1

`allocate-resources` transitions to `player_turn` by directly updating `games.phase` without calling `drawCardsForPlayer`. Initial hands were dealt by `start-game` at RAM=4. After a +2 RAM allocation (RAM=6), the first player still has 4 cards, needs 6. No code path draws the extra 2 for them — `drawCardsForPlayer` is only called in `advanceTurnOrPhase` (when advancing to the NEXT player), never for the first player of round 1.

### Why did draw-cards.spec.ts not catch this?

The test intentionally avoids both gaps:
- Uses `advanceToPlayerTurnNoBump` which allocates **zero CPU/RAM** — first player has RAM=4, hand=4, nothing to draw. Gap B is invisible.
- Plays with CPU=1, ≤1 card per turn — explicitly noted in the test: "CPU=1 with ≤1 card played → virus count = 0 → no virus resolution phase." Gap A is invisible.

The test verifies only the `end-play-phase` → `advanceTurnOrPhase` → round-2-draw path. Neither the virus path nor the allocation-bump path are covered.

## Item 2 — Mission progress not updating (Category A)

**User report:** Played Compute/Data cards across multiple turns; progress bar stayed at 0.

**Root cause:** `active_mission` is not included in the 3s polling loop in `GameBoard.tsx`. The poll only fetches `games` and `players`:

```typescript
const [{ data: g }, { data: p }] = await Promise.all([
  supabase.from("games").select("*").eq("id", gameId).single(),
  supabase.from("players").select("*").eq("game_id", gameId),
]);
// active_mission is absent
```

`active_mission` updates (`compute_contributed`, `data_contributed`, etc.) are delivered only via the Realtime `UPDATE` subscription. Per SESSION_NOTES, Realtime is unreliable; the same note acknowledges `game_log` may be missing when Realtime is broken. `active_mission` has the same weakness. A single Realtime hiccup leaves the mission board permanently stale for the session (no recovery path in the poll).

## Item 3 — Virus placement UI missing / resolution confusion (Category A + B)

**Category A — virus placement UI never built:**

`place-virus` backend (Phase 7) exists and accepts `{ card_ids, override_player_id }`. `PlayerTurn.tsx` has no UI to call it. AIs currently cannot place any cards into the virus pool during their turn. The pool that gets shuffled into the main pool is always empty (`pending_viruses` table stays empty). The only virus pool cards are the 4 face-down cards dealt at `start-game`.

**Category B — manual Resolve button (expected design):**
VirusResolution requires one click per card. This is deliberate: it lets players observe each resolution step. Not a bug.

**Category B — "Compute" as a virus card (expected):**
The virus pool is seeded with 4 random cards from the main deck. Progress cards (Compute, Data, Validation) can land in the pool. When resolved, they fall through to the `default` case in `applyVirusEffect` (logs "Virus resolved: compute.", no game effect). This is correct per spec: "Good AIs sacrifice Progress cards to dilute the pool."

## Item 4 — No log entry / no targeting UI (Category B)

**Log entries missing:** `resolve-next-virus.applyVirusEffect` writes a `game_log` INSERT for every card type, including the `default` case. The `GameLog` component and Realtime subscription are both in place. The entries exist in the DB. Not seeing them in the UI is the pre-Phase-11 Realtime-only gap (same root cause as Item 2 — no log polling). Expected.

**No secret-targeting UI:** Correct. Targeting is only triggered by: `process_crash`, `memory_leak`, `resource_surge`, `cpu_drain`, `memory_allocation`. The resolved card was `compute` (progress type, default case), which does not trigger `secret_targeting` phase. The private chat visible to misaligned AIs is always shown during `player_turn` — it is not targeting-specific.

## Item 5 — Cannot see own alignment (Category A)

**User report:** As AI player, no indication whether Aligned or Misaligned.

**Root cause:** `PlayerRoster.tsx → roleDisplay()` returns `"AI"` for both `aligned_ai` and `misaligned_ai`. There is no other UI surface showing own alignment.

Dev-mode partial workaround: `DevModeOverlay` PlayerSwitcher buttons show `"A"` (aligned) or `"M"` (misaligned) labels — but these are small, opacity-50, and only present in dev mode. A real-game player would have zero indication of their alignment.

The RLS policy allows each player to read their own `role` field at all times, so the data is available — it's a display-only gap.
